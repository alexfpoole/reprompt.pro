Lessons from real sessions

Atomic writes: Race conditions wiped a shared tracker file twice before I learned to write-to-temp-then-rename.

Baked-in test preferences ↓
  • No weak tests — every assertion must test something meaningful
  • No hidden failures — tests must fail visibly, not silently pass
  • Work around external dependencies — avoid fragile tests dependent on external services
  • Batch similar conditions — test multiple form error conditions in one test rather than restarting scaffolding for 10 near-identical tests
  • Idempotent tests — any selection can run in any order

Reuse: tests → user documentation

The same Playwright test suite became the foundation for auto-generating user documentation. Screenshots were captured during test runs.

Deterministic vs non-deterministic moment: Claude and Gemini were both unreliable at cropping screenshots to show only modals. The non-deterministic approach failed repeatedly. So we vibe-coded a programmatic solution that sampled colours in the screenshot to determine modal edges and crop accurately every time. Deterministic won.

Fan-out in action — subagents searching a large codebase for a specific code smell
Why this architecture ↓
  • Each subagent gets a strict, narrow prompt: enumerate dependencies, document I/O for public endpoints, record types
  • Each agent writes to its own file — multiple agents writing to one file causes data loss (they overwrite each other)
  • Reverse fan-out: collector agents combine outputs into summaries, working around context limits and file size constraints
  • The I/O map acts as a contract — if post-refactor I/O matches the spec, the refactoring is safe
  • Confidence comes from the enumeration being methodical, not from AI “understanding” the code holistically

Deterministic vs Non-Deterministic
The most important skill in AI-assisted development is knowing when to choose each approach.

Deterministic (scripts, programmatic logic):
Process automation, image cropping, fan-out documentation

Non-deterministic (headless AI, slash commands):
Email generation, classification, web design, strategy

The rule: same correct answer every time → deterministic. Benefits from variation or novel reasoning → non-deterministic, but sandboxed.

Fan-Out / Reverse Fan-Out
A reusable architecture for AI work across large codebases.

Fan-out: N parallel subagents, strict prompts, each writes to its own file.

Reverse fan-out: Collector agents combine outputs, working around context limits.

Key discovery: multiple agents writing to one file = data loss. Each agent must have its own output.

Iterative Refinement
AI output improves dramatically when you accumulate and encode preferences over time.

Tests: “no weak tests, no hidden failures, idempotent, batch conditions”

Emails: Streisand effect, atomic writes, end-to-end validation

These preferences are the real IP. The AI is the tool. The judgement is the value.

Where AI Falls Short
Specific, honest examples. Not theory.
  • Over-exploration: launches subtasks when told to just read a file
  • Verbosity: asks questions instead of making reasonable assumptions
  • Compounding regressions: fixes create new bugs downstream
  • Perspective confusion: addresses me as customer, not business owner
  • Streisand effect: “don’t mention X” makes X appear more
  • Context limits: large codebases need architectural workarounds
